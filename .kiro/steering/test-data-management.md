# Test Data Management

## Rules

- Never hardcode product names, card numbers, or credentials in test bodies or page methods
- All data shapes are consistent ‚Äî every data source uses identical keys
- Credentials live only in `utils/testUsers.ts`
- Worker-to-user mapping is deterministic ‚Äî never access `siteConfig.users[hardcodedIndex]`

## Data Sources ‚Äî Decision Table

| Situation | Source | Import |
|---|---|---|
| One test, one fixed dataset | `clientSiteSingleData.json` | `import data from '../../utils/clientSiteSingleData.json'` |
| Same test, multiple products | `clientSiteMultipleData.json` | `import dataSet from '...'` + `for...of` |
| Test uses `{ page, data }` destructuring | `fixture.ts` `data` fixture | automatic via custom `test` |
| API payload, mock body, intercepted URL | Inline `const` at top of spec file | none |
| Unique value per run (registration email) | `Math.random()` inline | none |

## Data Shape

All three file-based sources share the same keys:

| Key | Example value |
|---|---|
| `productName` | `"iphone 13 pro"` / `"ZARA COAT 3"` |
| `creditCardNumber` | `"4542 9931 9292 2293"` |
| `name` | `"Arya S Kumar"` / `"Anshika"` |
| `cvv` | `"452"` |
| `couponValue` | `"rahulshettyacademy"` |
| `countryName` | `"India"` / `"Cuba"` |

When adding new test scenarios requiring additional fields, add them to all three sources and update `TestData` type in `fixture.ts`.

## Parameterized Test Pattern

```typescript
import dataSet from '../../utils/clientSiteMultipleData.json'

for (const data of dataSet) {
    test(`Purchase ${data.productName} with other product`, async ({ page }, testInfo) => {
        // ...
    })
}
```

Include a data-driven identifier in the name so failures are immediately traceable.

## Inline Constants (spec-scoped data)

## Worker and User Mapping

```
siteConfig.users[0] ‚Üí Worker 0 ‚Üí storage/client/user-0.json (aryas1@test.com)
siteConfig.users[1] ‚Üí Worker 1 ‚Üí storage/client/user-1.json (aryas2@test.com)
...
siteConfig.users[5] ‚Üí Worker 5 ‚Üí storage/client/user-5.json (aryas6@test.com)
```

Modulo cap applied in `fixture.ts` and `storageHelper.ts`: `workerIndex % siteConfig.workers`.

`siteConfig.workers` must always be ‚â§ `siteConfig.users.length`.

## Storage State Files

`storage/client/user-N.json` contains `localStorage.token` and `localStorage.userId`.

- Generated by `global-setup.ts` ‚Äî never create or edit manually
- Read by `fixture.ts` (browser context) and `storageHelper.ts` (token/userId for API calls)
- Not committed to git ‚Äî regenerated fresh on every CI run

## What Not to Do

- Don't hardcode product names, CVV, card numbers in test or page files
- Don't use real financial data ‚Äî all values are test-environment fakes
- Don't commit `storage/` files ‚Äî they contain live JWT tokens
- Don't share one user across multiple parallel workers
